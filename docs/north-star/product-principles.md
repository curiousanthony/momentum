# Product Principles

## Core Design Principles

- `Make invisible progress visible`: the dashboard exists to surface meaningful patterns, not just counts
- `Optimize for competence first`: users should feel they are becoming better with Cursor
- `Support momentum without guilt`: consistency matters, but recovery matters too
- `Connect behavior to outcomes`: activity is only useful if it helps users build and ship more effectively
- `Prefer self-comparison over social comparison`: compare users to their past selves before comparing them to anyone else
- `Use game mechanics with restraint`: the product should feel gameful, not toy-like

## Product Pillars

- `Progression`: stage movement, mastery, milestones, and unlocks
- `Insight`: reflection, behavior patterns, and interpretable feedback
- `Momentum`: streaks, comeback loops, and habit support
- `Outcome`: project progress and signals tied to real work
- `Memory`: durable context for users and future agents

## Anti-Goals

- shallow point farming
- badge spam with no behavioral meaning
- manipulative loss pressure
- public leaderboards as a central mechanic
- feature clutter that obscures the core user promise

## Decision Filters

Before adding a feature, ask:

1. Does this teach the user something valuable about how they work with Cursor?
2. Does this increase competence, momentum, or outcome clarity?
3. Does this feel credible for serious engineers and ambitious builders?
4. Would this still be worth keeping if all points and badges disappeared?

If the answer to most of these is no, the feature likely does not belong.
